# Thermo Mining 仓库介绍

这是一份面向 PPT 展示的仓库介绍文稿，基于当前仓库实现整理，适合直接摘取为项目背景、架构说明、工具清单和阶段成果说明。

## 1. 项目定位

`Thermo Mining` 是一个面向嗜热蛋白挖掘的生物信息学工程仓库。它的目标不是只提供几个独立脚本，而是把蛋白筛选、聚类压缩、热稳定性预测、语义召回、结构预测、结构相似性确认、结果汇总和服务端任务管理整合为一套可运行、可恢复、可审计的完整平台。

从工程形态上看，这个仓库同时包含两部分：

- 一条面向蛋白候选挖掘的核心流水线
- 一个面向 Linux 服务器部署的轻量控制平面

前者负责真正的计算流程，后者负责规划任务、确认执行、监控运行状态、查看产物，并通过统一配置把算法流程和服务端运维连接起来。

## 2. 项目要解决的问题

在高温环境微生物研究中，研究者通常希望从大量候选蛋白中尽快筛出更可能具有热稳定性、结构合理性和进一步研究价值的目标。困难在于：

- 候选蛋白数量往往很多，直接做高成本结构分析代价太高
- 单一模型很难覆盖“广义嗜热蛋白”这一边界并不完全清晰的目标
- 实际计算通常发生在 Linux 服务器上，需要长任务托管、断点续跑和产物回溯
- 输入来源并不总是统一，可能来自蛋白文件、contigs，甚至上游 reads 流程

因此，这个仓库的核心思路不是依赖某一个模型“一步到位”，而是采用多阶段串联策略，先用便宜步骤压缩候选空间，再把更昂贵的结构类工具留给更高价值的候选集合。

## 3. 仓库当前提供的核心能力

当前仓库已经具备以下几类能力：

- 支持从 `proteins`、`contigs`、`paired_fastq` 等不同输入形态启动任务
- 支持标准蛋白挖掘主线，也支持新增的 `seeded_proteins` 种子引导挖掘模式
- 通过统一 stage 目录管理每一步输出，并写入 `DONE.json`、`scores.tsv`、报告文件等中间产物
- 支持 `resume` 机制，允许在输入和参数一致时跳过已完成阶段
- 提供 `thermo-mining` CLI，用于本地或服务端批量执行
- 提供基于 `FastAPI` 的单用户控制台，用于计划生成、任务确认、运行监控和结果查看
- 使用 `tmux` 托管长任务，使网页断开后后台任务仍能继续执行
- 提供 `server_probe.py`，用于在目标 Linux 服务器上探测工具、Conda 环境和部署候选路径

## 4. 核心流程概览

### 4.1 标准蛋白挖掘流程

普通 `proteins` 模式下，主线流程可以概括为：

`prefilter -> MMseqs2 cluster -> TemStaPro -> ProTrek -> structure predict -> Foldseek -> report`

其设计思路是：

- 先过滤过短、过长或成分异常的蛋白序列
- 再通过聚类减少冗余，压缩候选规模
- 使用热稳定性模型进行第一轮筛选
- 使用语义召回模型做进一步排序
- 对保留候选进行结构预测
- 用结构相似性工具做最终确认
- 最后输出 tier 化结果和汇总报告

### 4.2 Seeded Mining 流程

当前仓库还加入了 `run-seeded` 模式，用于“已知种子蛋白引导目标蛋白库挖掘”的场景。其流程为：

`prefilter -> MMseqs2 cluster -> seed sequence recall -> seed structure recall -> seed merge -> TemStaPro -> ProTrek -> structure predict -> Foldseek -> report`

这个模式的意义在于：

- 用户可以提供一个 `seed_faa` 作为已知参考
- 再提供一个 `target_faa` 作为待挖掘目标库
- 系统先从目标库中召回与种子在序列或结构上相近的候选
- 然后只对召回到的目标集合继续进行热稳定性与结构确认流程

这样做比直接把种子文件本身当作候选集更合理，更适合定向挖掘和同源扩展场景。

## 5. 仓库的工程结构

从代码结构看，这个仓库可以分成几层：

### 5.1 CLI 与运行入口

入口位于 `src/thermo_mining/cli.py`，当前已经支持：

- `thermo-mining run`
- `thermo-mining run-seeded`
- `thermo-mining serve`
- `thermo-mining run-job`

这说明仓库既支持直接跑流水线，也支持启动服务端控制平面，还支持由后台 job runner 读取落盘任务目录后执行。

### 5.2 Pipeline 执行层

核心执行逻辑位于 `src/thermo_mining/pipeline.py`。这一层负责：

- 根据 bundle 类型选择 stage 顺序
- 为每个 stage 创建目录
- 调用各个步骤实现
- 处理 resume 跳过逻辑
- 合并分阶段得分
- 生成最终报告

### 5.3 Step 适配层

`src/thermo_mining/steps/` 目录封装了具体的阶段逻辑，包括：

- `prefilter`
- `mmseqs_cluster`
- `temstapro_screen`
- `protrek_bridge`
- `structure_predict`
- `foldseek_client`
- `seed_sequence_recall`
- `seed_structure_recall`
- `seed_recall_merge`
- `rerank`

这一层是具体生信工具与 Python 工程之间的桥梁。

### 5.4 控制平面

`src/thermo_mining/control_plane/` 和 `src/thermo_mining/web/` 实现了轻量控制平面，主要负责：

- 文件浏览
- 输入 bundle 组织
- 计划生成与校验
- 单任务锁控制
- `tmux` 生命周期管理
- `runtime_state.json` 状态维护
- 网页路由与 API 输出

### 5.5 配置与部署辅助

仓库通过 `config/platform.example.yaml` 和 `settings.py` 统一管理：

- LLM 配置
- 运行目录
- 服务地址
- 日志路径
- 外部工具路径
- 默认阈值与参数

这让同一套代码既可以在本地开发环境运行，也可以迁移到 Linux 服务器执行。

## 6. 仓库中各个核心工具的简要介绍

这里的“工具”既包括生物信息学计算工具，也包括平台运行和交互层使用的工程工具。

### 6.1 fastp

`fastp` 用于原始双端测序数据的质量控制和预处理。在本仓库中，它主要出现在 `paired_fastq` 输入链路中，负责对 reads 做前置清洗，为后续组装提供更稳定的输入。

### 6.2 SPAdes

`SPAdes` 是常见的组装工具。在本仓库的设计中，它用于把 `paired_fastq` 数据组装为 contigs，是从原始 reads 进入下游蛋白挖掘流程的重要中间步骤。

### 6.3 Prodigal

`Prodigal` 用于 ORF 预测和蛋白编码区识别。在仓库的整体链路中，它承担从 contigs 生成蛋白序列的角色，让上游装配产物可以转化为后续挖掘阶段真正使用的 `proteins.faa`。

### 6.4 MMseqs2

`MMseqs2` 是本仓库中非常关键的序列层工具，主要承担两项工作：

- 对候选蛋白做聚类压缩，减少冗余
- 在 seeded 模式下进行基于序列相似性的 seed recall

它的价值在于运行成本相对较低，适合在流程前段快速缩小候选空间，是整个多阶段策略中的“高效预压缩器”。

### 6.5 TemStaPro

`TemStaPro` 是热稳定性预测相关工具，在本仓库中用作热稳定性初筛步骤。它不是最终决策者，但能较早地把更不可能的候选剔除，从而为后续更昂贵的步骤节约成本。

### 6.6 ProTrek

`ProTrek` 在本仓库中承担语义召回或语义重排角色。仓库会用预设的查询语句，例如 `thermostable enzyme`、`heat-stable protein` 等，对候选蛋白进行语义层排序。它的价值在于引入了超越纯序列或纯结构匹配之外的表征能力。

### 6.7 ColabFold / LocalColabFold

仓库通过 `colabfold_batch` 调用结构预测能力，用于把保留下来的候选蛋白序列转为结构文件。这一步在普通流程中用于后续 `Foldseek` 确认，在 seeded 模式中也用于 seed structure recall。

它是整条流程里相对昂贵的步骤之一，因此仓库设计强调应尽可能在前面先压缩候选数量。

### 6.8 Foldseek

`Foldseek` 用于结构相似性搜索。在本仓库中有两种主要用途：

- 在主线末端对候选蛋白做结构确认
- 在 seeded 模式中对种子和目标代表结构做召回

它承担的是“结构证据层”的角色，通常比单纯序列方法更接近真实功能与折叠相似性的验证需求。

### 6.9 tmux

`tmux` 不是生信算法工具，但却是这个仓库部署策略中的关键工程工具。控制平面会用 `tmux` 启动和托管后台任务，使得：

- 用户关闭浏览器后任务仍可继续
- 服务器端长时间运行更稳定
- 必要时可以直接 attach 到会话进行调试

因此，`tmux` 是连接网页控制台与后端长任务执行的核心运行支点。

### 6.10 FastAPI

`FastAPI` 是仓库控制平面的 Web 框架。它负责：

- 提供网页入口
- 暴露文件系统、计划、运行状态等 API
- 把控制逻辑组织成可调用的服务接口

FastAPI 让这个仓库从“可跑的脚本集合”进一步升级为“可交互的平台”。

### 6.11 OpenAI 兼容 LLM 接口

仓库配置中包含 OpenAI 兼容接口设置，用于把用户自然语言转换成结构化执行计划。这里的 LLM 不是直接执行 shell 命令，而是负责：

- 解析用户意图
- 生成受限的执行计划
- 辅助说明当前任务状态

也就是说，LLM 在这个仓库里更像“计划器”和“解释器”，而不是自由代理。

### 6.12 server_probe.py

`scripts/server_probe.py` 是一个非常实用的部署前探测工具。它会在目标 Linux 服务器上检查：

- 常用工具是否存在
- Conda 环境是否可用
- ProTrek、Foldseek 等关键路径是否可定位
- 数据目录、运行目录和日志目录的候选位置

它最终会产出结构化 JSON、配置草稿 YAML 和文本摘要，帮助把“仓库代码”顺利落到“真实服务器环境”上。

## 7. 这个仓库的设计亮点

相较于只关注算法或只关注界面的项目，这个仓库的特点在于它把科学计算流程与工程可运维性放在同一层面考虑。

### 7.1 多阶段串联，而非单点依赖

仓库并不假设某一个模型可以直接解决所有问题，而是采用“预过滤 + 压缩 + 热稳定性筛选 + 语义召回 + 结构确认”的组合式路径。这种设计更符合真实研究中的成本控制和证据整合逻辑。

### 7.2 可恢复、可审计

每个阶段都有自己的目录和状态产物，包括 `DONE.json`、`scores.tsv`、结构清单、运行状态文件等。这样做的好处是：

- 可以断点续跑
- 可以定位失败位置
- 可以回溯每一步的输入与结果

### 7.3 控制平面和计算流程统一

CLI 和 Web 控制台共用同一套配置与运行模型，不是两套完全分裂的系统。这样既方便脚本化运行，也方便给用户提供更友好的交互界面。

### 7.4 面向服务器部署

从 `tmux`、路径浏览、配置模板、server probe 到控制台运行模型，这个仓库明显不是只为了本地实验，而是围绕 Linux 服务器部署场景设计的。

### 7.5 已支持种子引导挖掘

新增的 seeded mining 说明这个仓库已经开始从“通用筛选流程”走向“带先验知识的定向挖掘流程”，这让它在科研场景下更有扩展空间。

### 11.1 用一句话理解 ProTrek

如果只用一句话解释，`ProTrek` 可以理解为：**一个把蛋白序列、蛋白结构和功能文字说明放进同一个坐标系里的蛋白检索模型。**

一旦这三类信息被映射到同一个空间里，就可以做：

- 用文字描述找蛋白
- 用一个已知蛋白找一批相近蛋白
- 用结构信息辅助发现可能功能相关的候选

它的核心价值，不是简单地“做一次比对”，而是让序列、结构和功能注释这三种原本分散的信息能够互相对齐、互相检索。

### 11.2 ProTrek 的基本原理

可以把传统序列检索理解成“看两个蛋白字面上像不像”，而把 `ProTrek` 理解成“看两个蛋白在更高层次上是不是同一类东西”。

它训练时同时学习三种信息：

- 蛋白序列
- 蛋白结构
- 蛋白功能文字描述

模型的目标，是让原本互相对应的序列、结构和功能描述，在数学空间里彼此靠近；让不相关的样本彼此远离。这样做的结果是：即使两个蛋白在序列上不算很像，只要它们在结构或功能意义上更接近，`ProTrek` 也有机会把它们检索到一起。

可以把它想象成一个同时会“读序列、看结构、理解注释”的系统。传统工具往往擅长其中一个维度，而 `ProTrek` 的特点是把这三维信息联通起来。

### 11.3 ProTrek 能解决什么问题

`ProTrek` 特别适合解决下面这类问题：

- 我只有一个已知蛋白，想找更多可能同类或近似功能的蛋白
- 我只有一段功能描述，想快速找出可能相关的蛋白候选
- 我不想只依赖序列同源性，因为有些功能相关蛋白在序列上已经比较远
- 我面对的是非常大的数据库，需要一种比逐条人工筛更高效的候选召回方式

这对嗜热蛋白或 Cas 相关蛋白挖掘尤其重要。因为很多研究目标并不是一个单靠序列保守位点就能完全定义清楚的类别。研究者更关心的往往是：它是否像一类值得继续验证的蛋白，而不是它是否和模板蛋白逐位点都很像。

### 11.4 ProTrek 在当前项目里是怎么用的

在 `all-ai-bio` 当前的实际工作里，`ProTrek` 不是单独给出最终结论的工具，而是整个筛选流程中非常关键的一层“高价值候选重排器”。

整体思路是：

1. 先用 `prefilter`、`MMseqs2`、`TemStaPro` 等成本较低的步骤，尽量缩小候选空间。
2. 再把压缩后的候选交给 `ProTrek`，让它做更有信息量的语义检索或重排。

最近在做的 Cas seed mining 就是一个非常具体的例子。我们先把 Cas12f、Cas12a、TnpB、Cas9、IscB 等家族的代表片段整理成一个 seed FASTA；然后用 `ProTrek` 给大蛋白数据库建立索引；再把这些 seed 序列编码成向量，到索引里寻找“最接近”的蛋白；最后把命中的结果去重、按分数排序，输出成候选表。对使用者来说，这相当于把“我手里已有的一小批已知参考”扩展成“一个更大的潜在同类候选集”。

### 11.5 ProTrek的角色

如果说 `MMseqs2` 更像高效的粗筛工具，`Foldseek` 更像结构层面的确认工具，那么 `ProTrek` 在这个体系里承担的就是“把真正值得追的候选提前找出来”的角色。它的重要性不只在于速度，更在于它能帮助我们跨过“低序列一致性但可能功能相关”的那道门槛。可以把它理解成这样：`ProTrek` 不是替代实验，也不是替代结构验证；它做的是在海量蛋白里先帮我们更聪明地挑人，把更像目标、也更值得做后续验证的候选优先送到前面。这样，后面的结构分析、功能验证和实验资源投入就会更集中，也更有命中率。
